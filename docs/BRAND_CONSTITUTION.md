# Ironflyer Brand Constitution

## Position

Ironflyer is the Finisher OS for AI-built software.

The market already understands prompt-to-app. Ironflyer owns what comes next:
reviewable patches, blocking gates, real runtime evidence, budget control,
audit trails, and deploy readiness.

## Promise

Build the product. Prove it is done.

Every surface should answer:

- What is being built?
- What is blocked?
- What evidence says it is safe to ship?
- What can I approve from mobile right now?

## Competitive Wedge

Use the patterns that work for v0, Bolt, Replit, and Windsurf:

- one prompt entry
- instant visual feedback
- templates that reduce blank-page anxiety
- deploy path visible early
- mobile access
- integrations and repo sync

Own the gap they leave open:

- gates that block unfinished work
- patch-first AI edits instead of mystery writes
- transparent provider cost and margin control
- real Linux runtime ownership
- visual QA targets and pixel-diff evidence
- audit-grade memory and event history

## Visual System

Primary mode is dark. Light mode exists for docs, export, and bright work
contexts.

Palette:

- Graphite: `#07090d`
- Slate: `#10151d`
- Paper: `#f5f2ea`
- Mint signal: `#53ffbd`
- Cyan evidence: `#5bc8ff`
- Amber blocker: `#ffd166`
- Ember danger: `#ff7a45`

Use color as status and evidence, not decoration.

## Typography

Display type is blunt, uppercase, and compact. It should feel like a command,
not a slogan. Body type should be calm, readable, and technical.

Never use negative letter spacing. Never make compact dashboard headings hero
sized.

## Logo

The mark is a forward finisher glyph: F-shape, gate rails, and an outbound
arrow. It should read as control, movement, and completed work.

Use:

- dark mark on paper for light surfaces
- paper mark on dark surfaces
- mint/cyan only as the signal pair

## Product UI Rules

- First screen must be usable product, not a marketing brochure.
- Prefer dense but calm control surfaces over decorative cards.
- Cards are for repeated objects, modals, and framed tools only.
- Dashboard and workspace screens must expose gates, patches, runtime, budget,
  and deploy state without scrolling on desktop.
- Mobile must prioritize approvals, blockers, preview, patch risk, and run
  status.

## Asset Rules

Approved primary assets:

- `/brand/data-flow.jpg` for data/build flow and hero depth
- `/brand/motion/finisher-os-hero-loop.mp4` for full-bleed hero motion
- `/brand/motion/gate-evidence-flow.mp4` for gates, risk, and evidence sections
- `/marketplace/output-ref/fx.png` as workspace/product-surface reference
- `/brand/icon-ai-code.png`, `/brand/icon-repository.png`,
  `/brand/icon-console.png`, `/brand/icon-api.png` for primitive cards
- `/brand/glyphs/*.svg` for special capability icons in product surfaces
- existing `/templates/*.jpg` thumbnails for revenue-ready starts

Avoid generic developer/person illustrations as primary brand visuals. They
can be used in docs or empty states only when the message benefits from them.

The source of truth for web usage is `apps/web/lib/brand-assets.ts`. New
assets must be added there with source path, intended use, and guidance before
they appear in UI code.

## Voice

Short, direct, evidence-led.

Say:

- "Deploy blocked: security repair queued."
- "Patch risk low. Preview responsive. Cost inside cap."
- "Build the product. Prove it is done."

Avoid:

- vague AI magic
- "10x" without proof
- overpromising autonomous delivery without gates
- generic "beautiful apps in minutes" copy

## Revenue Narrative

Ironflyer should help the company reach million-shekel monthly revenue by
selling finished outcomes:

- high-ticket agency delivery
- productized MVP packages
- enterprise internal tooling
- governed AI build factory
- private-cloud and audit-led plans

Every page should make the buyer feel that Ironflyer reduces delivery risk,
protects margin, and makes finished software easier to sell.
